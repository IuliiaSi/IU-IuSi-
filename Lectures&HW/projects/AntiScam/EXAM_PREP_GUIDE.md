# Экзаменационный гайд по приложению «Честная оценка» (AntiScam)

Этот документ написан как шпаргалка для защиты проекта.
Цель: чтобы даже человек с минимальным опытом в IT мог объяснить, как работает приложение, зачем оно нужно, как его запускать и какие в нём меры безопасности.

---

## 1) Что это за приложение простыми словами

`Честная оценка` — это помощник для автовладельца.
Он помогает понять, какие работы из списка автосервиса действительно важные, а какие можно обсудить или отложить.

Главная идея:
- не спорить «на эмоциях» с сервисом;
- получить второе мнение;
- увидеть приоритеты работ в понятной форме.

---

## 2) Как объяснить проект за 30 секунд (готовый ответ)

«Это MVP-приложение для второго мнения по рекомендациям автосервиса. Пользователь входит по email, вводит данные авто и список работ, а система группирует работы по приоритету и сохраняет историю в Supabase. Архитектура разделена на frontend (Vue) и backend (NestJS). Мы добавили базовую безопасность: `/success` больше не выдаёт доступ, paid-права проверяются на сервере, есть cooldown 3 секунды и подтверждение “Я не робот” перед основным действием».

---

## 3) Для кого продукт и какую проблему решает

### Целевая аудитория
- автовладельцы без технической экспертизы;
- люди, которым сложно оценить рекомендации сервиса;
- пользователи, которые хотят «перепроверить» смету спокойно и структурно.

### Проблема
- пользователь получает список работ и не понимает срочность;
- есть страх переплаты за ненужные услуги;
- сложно вести историю прошлых разборов.

### Решение
- приложение сортирует работы по группам приоритета;
- объясняет каждую позицию простым языком;
- хранит историю записей пользователя.

---

## 4) Что именно умеет MVP

- регистрация/вход по email и паролю (Supabase Auth);
- ввод данных автомобиля;
- выбор метода:
  - ручной выбор работ,
  - загрузка файла (демо-сценарий),
  - пример результата;
- показ результата с приоритетами;
- переход к paywall-экрану;
- история записей (`entries`) для пользователя с подтверждённым paid-доступом.

Важно: это MVP, поэтому платёжный поток демонстрационный, а не полноценный интегрированный эквайринг.

---

## 5) Архитектура без сложных терминов

Проект состоит из двух частей:

1. **Frontend (`frontend`)**  
   Что видит пользователь: экраны, кнопки, формы.

2. **Backend (`backend`)**  
   Сервер, который:
   - принимает запросы от frontend,
   - проверяет пользователя,
   - работает с Supabase,
   - применяет защитные проверки.

3. **Supabase**  
   Внешний сервис, где:
   - авторизация пользователей,
   - таблицы с данными (`entries`, `cars`, `user_access`),
   - политики доступа (RLS).

## 7) Путь пользователя (сценарий по шагам)

1. Открывает приложение.
2. Входит или регистрируется.
3. Вводит данные авто.
4. Выбирает способ добавления работ.
5. Делает анализ.
6. Смотрит результат по группам.
7. Попадает на paywall.
8. Может перейти на `/success` (информационный шаг).
9. Возвращается в приложение.
10. Доступ к платным данным определяется сервером, а не фактом перехода на страницу.

## 8) Объяснение ключевых экранов для защиты

### `/login`
- вход/регистрация;
- получает access/refresh токены;
- сохраняет сессию.

### `/manual`
- основной action MVP;
- пользователь выбирает работы;
- перед запуском обязателен чекбокс «Я не робот»;
- есть cooldown 3 секунды.

### `/analysis`
- показывает этапы анализа.

### `/result`
- итоговые группы приоритетов;
- блок истории (`entries`) доступен только при paid-доступе.

### `/paywall`
- коммерческий экран с предложением доступа.

### `/success`
- только сообщение «Оплата почти завершена»;
- не меняет данные в БД;
- кнопка «Вернуться в приложение».

## 9) Что было сделано по безопасности MVP

Ниже — ровно те меры, которые обычно спрашивают на проверке.

### 9.1 Защита `/success`
- `/success` не выдаёт paid-доступ;
- `/success` не пишет `paid = true`;
- `/success` ничего не изменяет в Supabase/БД.

### 9.2 Серверный источник правды для paid
- добавлен endpoint `GET /api/access/me`;
- backend читает paid/role из таблицы `user_access`;
- frontend только отображает статус, но не решает сам.

### 9.3 Убрано доверие к клиенту для платной функции
- доступ к истории (`entries`) проверяется на сервере (`requirePaidAccess`);
- прямой переход по URL не даёт обойти проверку.

### 9.4 Cooldown 3 секунды
- на frontend: блокируется повторный запуск основного действия;
- на backend: дублирующая проверка частоты;
- текст ошибки: `Подождите немного`.

### 9.5 Подтверждение «Я не робот»
- на frontend обязательный чекбокс;
- на backend DTO требует `humanConfirmed = true`.

### 9.6 Секреты
- на frontend не используются приватные ключи;
- backend использует env-переменные (`SUPABASE_URL`, `SUPABASE_ANON_KEY`);
- service-role ключ не выносится в клиент.

### 9.7 Базовые RLS-политики
- отдельный SQL-файл: `sql/mvp_security.sql`;
- таблица `user_access` с RLS;
- клиенту запрещены прямые `insert/update/delete` для `paid/role`.

## 11) Как запускать проект (коротко)

### Вариант 1: Replit
1. Добавить Secrets:
   - `SUPABASE_URL`
   - `SUPABASE_ANON_KEY`
2. Нажать Run.

### Вариант 2: Локально
- backend:
  - `cd backend`
  - `npm install`
  - `npm run start:dev`
- frontend:
  - `cd frontend`
  - `npm install`
  - `npm run dev`

### Вариант 3: Docker
- в корне AntiScam:
  - `docker-compose up --build`

## 13) Самые частые вопросы на защите и готовые ответы

### Вопрос: Почему вы не ставите paid через frontend?
Ответ: потому что frontend нельзя считать доверенной стороной. Любой может подделать клиентский запрос. Поэтому paid-статус хранится и проверяется на сервере через `user_access`.

### Вопрос: Зачем вообще страница `/success`, если она ничего не меняет?
Ответ: это безопасная промежуточная UX-страница. Она информирует пользователя, но не даёт доступ без подтверждённого server-side статуса.

### Вопрос: Где защищён доступ к платной функции?
Ответ: на backend в сервисе `entries` через проверку paid-статуса (`requirePaidAccess`), а не только на уровне кнопок/экранов.

### Вопрос: Почему cooldown должен быть и на сервере, и на клиенте?
Ответ: только клиентский cooldown можно обойти скриптом. Серверный cooldown защищает API независимо от UI.

### Вопрос: Чекбокс «Я не робот» — это настоящий CAPTCHA?
Ответ: нет, это MVP-механика подтверждения действия по заданию. Для production лучше подключать полноценный challenge (например, Cloudflare Turnstile/reCAPTCHA) с server-side проверкой токена.

### Вопрос: Как вы защищаете данные пользователей?
Ответ: через RLS-политики в Supabase: пользователь видит только свои строки (`auth.uid() = user_id`).

### Вопрос: Можно ли через SQL API самому сделать себе admin?
Ответ: нет. В `user_access` клиенту запрещены прямые `insert/update/delete` для прав доступа.

### Вопрос: Почему вы не храните секреты во frontend?
Ответ: потому что фронтенд код доступен пользователю в браузере. Секреты должны быть только на сервере и в env.

### Вопрос: Где роль admin?
Ответ: роль хранится в `user_access.role` и возвращается backend’ом в `/api/access/me`. Сейчас реализован базовый слой ролей без усложнения RBAC.

### Вопрос: Какую часть вы считаете “основным действием” в MVP?
Ответ: запуск ручного анализа в `/manual` — это главное действие, которое пользователь может нажимать много раз подряд.

## 14) Словарь терминов для новичка

- **Frontend** — визуальная часть, которую видит пользователь.
- **Backend** — серверная часть, где происходят проверки и работа с БД.
- **API** — «дверь» между frontend и backend.
- **Supabase** — сервис для auth и базы данных.
- **RLS** — правила на уровне БД: кто какие строки может читать/менять.
- **Route (роут)** — URL-адрес страницы, например `/result`.
- **DTO** — формат входных данных запроса.
- **Cooldown** — минимальная пауза между повторными действиями.
- **Env-переменная** — секрет/настройка, которая хранится вне кода.

## 15) Типичные ошибки, которые важно не допустить на экзамене

1. Не говорить, что “перешли на `/success` = оплатили”.
2. Не утверждать, что frontend может безопасно хранить секреты.
3. Не путать UI-ограничение и server-side проверку.
4. Не забывать упомянуть RLS.
5. Не говорить, что чекбокс “Я не робот” — это полноценная антибот-система.

## 16) Короткий итог для преподавателя

Проект реализует рабочий MVP второго мнения для автосервиса и содержит базовый security-набор:
- защищённый `/success` без выдачи paid;
- server-side источник прав (`user_access`);
- серверная проверка платного доступа;
- cooldown 3 секунды;
- обязательное подтверждение «Я не робот»;
- базовые RLS-политики;
- отсутствие клиентских секретов.

Этого достаточно для учебного MVP с понятной логикой и объяснимыми компромиссами.
